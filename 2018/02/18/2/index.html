Codeforces Round 460 划水记

2018-02-18

第一次尝试给CF出题，感觉棒棒哒。特写一篇划水记来纪念这次神奇的经历。

造题阶段

焦急等待

大概就是有了一些小小的idea，觉得可以出在Div.2 CDE题了，于是就准备出一场CF。

CF对出题人是有限制的，大概是蓝名25场rated比赛，紫名15场，黄名5场，红名没有限制。出题需要在一个专门的地方把你的题目写好提交，最后由CF的coordinator审核你的题目，决定是否录用你的题目。

但值得注意的一点是，从提交到通过审核这个过程历时非常之久，因为提交Contest Proposal的人太多了。(我是17年8月提交的proposal，审核通过大概是在18年年初QAQ)

于是隔些时间就给coordinator发私信，结果人家基本不理我。OvO

于是焦急等待了好几个月。

希望曙光

终于在某一天发私信后，coordinator回复了我，回复了我，回复了我(重要的事情说三遍233)，并表示他已经对我的题目做出了comment。欣喜之余打开一看，发现基本上都是：

  • I think it is too straightforward even for D2A.
  • I’m sure I’ve seen this problem somewhere, so it has to be replaced.
  • I think there wasn’t exactly this problem, but the approach is widely-known, so it’s better to replace the problem.
  • …

此处手动添加一个”生无可恋”表情。

大概只有1.5道题通过了他的审核。(那半道题还是我给他解释了以后他才觉得OK的)

这两道题分别是后来的E和F。(想知道为啥有F的接着往下看吧233)

于是我回信说：“好吧，那我再回去造点题。”

再造题

其实通过审核的是Div.2 CE两题，这样我只需要造个D出来，基本上问题就不大了，因为AB是分分钟就可以yy出来的。

然后去翻了一些资料，希望找点灵感出来。大概是翻来翻去，找到了一个字符串题。然后魔改了一下(改了让求的东西)，觉得改完以后有个地方还是很喵的(其实也很蠢)，于是决定就这道题当D吧，提交上去又等了几个星期，最后

过审核了。

过审核了。

过审核了。

这道题是后来的D。

也是很开心了。

小插曲

现在有了CDE，还差AB。照理来说AB是好造的，然而造了几道，不是被说太难，就是被说太简单。

还有另外一个问题，就是C。这个coordinator认为C太难了，因为需要求逆元。(逆元这种东西不应该是个人就会的嘛)他建议我把求逆元那一部分去掉，就是手动把需要求逆元的地方挪到等式右边。但是我是不同意的，因为一我觉得那样写的等式实在是太难看了，二其实我觉得这样写更容易想到我给出的解法。(还有就是我懒)

接下来神奇的一幕发生了，由于我觉得我的题实在太水，再加之我多出了一道题，我决定这场比赛我出六道题。然后，C变成D，这样我就可以正大光明地维持C原来的样子，然后多一道题还可以让大家多玩一玩，顺便显得我的题没那么水，后来才发现这样也可以多一道hack题。

好处多多有木有！

当然coordinator也同意了。

于是现在的ABCDEF分别对应后来的ABCEDF，然后可以准备写题面造数据了。

准备阶段

必备小知识

CF的coordinator的ID是KAN，以后我就称他为KAN了。

KAN给我发了一个文件，这个文件告诉我应该如何造题：(以下为节选)

  • 题面
    • 题面不应过长，长度不应超过初始的文本框大小。
    • 在输入的变量后立即给出该变量的范围。
    • …
  • 数据
    • 如果可以，记得把所有的小数据全都加入test里。(这非常重要)
    • 每道题通常有20到100组测试数据。
    • Pretests应该有极限数据，确保爆int不能过的数据，还有一些随机生成的小数据。
    • Div.2 A应该有7-9组Pretests，Div.1 E应该有10-20组Pretests。
    • …
  • Checker和Validator
    • 请使用testlib.h。
    • …
  • …

这个文件里还说，之后所有要做的工作都是要在一个名曰polygon的平台上完成的，这个平台真的是非常awesome，有如下非常棒的特性：

  • 写题面，并生成为HTML或Latex格式。
  • 使用数据生成器生成数据，支持向生成器传参数。
  • 写各种程序，包括标算暴力等等，并且可以通过innovation验证程序的正确性，以及通过stress功能hack掉某些程序。
  • 每个人编辑的是远程数据的一份拷贝，只有在Commit Changes以后才会试图修改远程数据。这样就不用各种拷文件传文件辣！
  • 添加Issue，相当于一个讨论版。
  • …

总而言之非常方便非常爽。而且导出的pdf文件也非常好看有木有！

如何准备一道题

正式进入写题面&造数据阶段。那么准备一道题都需要写哪些东西呢？

Statement

你需要用英文写一个题面。题面要严格按照文件中所给的格式来写，支持Latex也是很棒棒的。

Checker

如果你可以应用内置的Checker，那么你直接选中那个Checker即可(表示全都是用的内置的哈哈哈哈哈)。否则你就要自己写一个了，写Checker时一定要用testlib，再者testlib用起来也方便。

Validator

这个东西是给hack用的。当比赛中有人试图hack某个程序时，会先让Validator来检查数据是否合法(符合题面要求)，如果Validator判定不合法，那就会直接丢出一个Invalid Input，就不会说把一堆奇奇怪怪的输入丢给标程最后RE了。另外你造数据的时候，polygon也会自动运行Validator来检查你造的数据是否合法，这也给数据加了一道安全锁。

Tests

造数据的地方。这里造数据有两种造法，一种是”Manual”，表示是手敲进去的数据；一种是”Script”，表示用数据生成器造的数据。手敲的数据就不用我多说了，关于用”Script”造数据，在这里你需要先用testlib(强烈建议)写一个数据生成器，然后上传到Files里，最后写一行类似脚本的东西，表示你用哪个数据生成器，给生成器传什么参数，这样就造好一组数据啦！

最后，你还需要再加一个Testset，命名为”pretests”，这里面的测试数据会被用做比赛中的Pretests，然后将tests中的前几组数据复制到pretests里面就可以啦！

Solution Files

这里可以加入一系列的Solution，包括”Main Correct Solution”(用于生成数据)，”Correct”(正确程序)，以及各种挂掉的程序，如”Wrong Answer”(答案错误)，”Time Limit Exceeded”(超出时限)等等。验题人的程序也会放在这里。

Invocations

在这里你可以把所有的Solution在任意的testset上跑一遍，如果出现了意料之外的情况(如标记为WA的AC了)，polygon也会用鲜明的颜色提示你出现问题了。

Stresses

在这里你可以卡某些程序，或者验证程序的正确性。你需要写一个脚本表示用哪个数据生成器生成数据，然后polygon会自动在后台运行所有的程序并对比输出结果。结果不同会用鲜艳的颜色来提示你出错了。
